# EVIDENCIA DE RECUPERACIÓN - FALLO 5: EL CORREO PERDIDO

## 📌 Descripción del Experimento
Se simula el apagado total del servicio de Notificaciones (despacho de correos de confirmación). Al ser un servicio no crítico para el negocio, el flujo central de compra en Reservas debe completarse con éxito, persistir la transacción en base de datos y retornar una respuesta exitosa degradada (Fallback) informando al cliente que el correo queda en cola.

---

## 💻 Comando de Inyección del Caos
```bash
# Apagar por completo el microservicio de notificaciones escalando sus replicas a 0
kubectl scale deployment/notificaciones-service --replicas=0
```

## 📈 Métrica de Éxito
* **Comportamiento Esperado:** La reserva del usuario se registra de forma exitosa. El cliente recibe HTTP 200 en lugar de un HTTP 500 debido a la caída de las alertas de correo.
* **Resultado Obtenido (JSON de Respuesta):**
  ```json
  {
    "status": "Reserva Completada",
    "reserva_id": "RES-1689729851000",
    "asiento_id": "asiento_5",
    "notificacion_estado": "Pendiente de reenvío en background"
  }
  ```
  *(Nota que el estado de la notificación cambió a "Pendiente de reenvío en background" de manera segura, sin colapsar el sistema de reservas).*
