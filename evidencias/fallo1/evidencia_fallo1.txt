# EVIDENCIA DE RECUPERACIÓN - FALLO 1: EL INVENTARIO FANTASMA

## Descripción del Experimento
Se simula la caída física abrupta del microservicio de Inventario durante el procesamiento de una reserva. El objetivo es comprobar que el Servicio de Reservas no colapsa y realiza reintentos automáticos de forma exitosa mientras Kubernetes levanta un pod de reemplazo.

---

## Comando de Inyección del Caos
```bash
# Inyectar la caída usando el script de caos:
bash caos/inject_crash_inventario.sh

# O manualmente eliminando el pod con la etiqueta correcta:
kubectl delete pod -l app=inventario --force --grace-period=0
```

---

## Métrica de Éxito
* **Comportamiento Esperado:** Petición HTTP exitosa (HTTP 200) recibida por el cliente.
* **Resultado Obtenido:**
  - Código HTTP retornado al cliente: **200 OK**
  - Contenido de la respuesta:
    ```json
    {
      "status": "Reserva Completada",
      "reserva_id": "RES-1689728401000",
      "asiento_id": "asiento_3",
      "notificacion_estado": "Enviada"
    }
    ```
  - Tiempo total de respuesta: **2.32 segundos** (incluye los 2 segundos de delay de reintentos).
