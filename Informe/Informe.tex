\section{Parte I: Diseño de la Arquitectura del Sistema}

\subsection{Descripción General del Sistema}
El sistema implementado corresponde a una plataforma distribuida de reservas de asientos para eventos. La arquitectura de microservicios está compuesta por 6 componentes fundamentales:
\begin{enumerate}
    \item \textbf{API Gateway:} Único punto de entrada perimetral que gestiona el enrutamiento y aplica control de tasa de peticiones (Rate Limiting).
    \item \textbf{Servicio de Reservas (Core):} Orquestador de la reserva. Contiene las reglas de negocio, lógica de reintentos, timeouts y el mecanismo de Circuit Breaker.
    \item \textbf{Servicio de Inventario:} Administra el catálogo de asientos y el stock disponible interactuando con su propia base de datos PostgreSQL.
    \item \textbf{Servicio de Pagos (Stub):} Simulador que procesa los cobros financieros y admite inyección dinámica de latencias altas (caos).
    \item \textbf{Servicio de Notificaciones (Stub):} Gestiona el envío de correos electrónicos. Admite apagados transitorios.
    \item \textbf{Base de Datos (PostgreSQL):} Motor de base de datos relacional independiente para el almacenamiento del estado persistente de Reservas e Inventario (patrón Database-per-Service).
\end{enumerate}

\subsection{Distribución en Clúster Kubernetes Multinodo}
Para cumplir con la tolerancia a fallos en infraestructura real, se desplegó un clúster Kubernetes multinodo local utilizando Minikube (\texttt{--nodes 2}). La distribución de los pods se forzó determinísticamente mediante reglas de afinidad y selectores de host (\texttt{nodeSelector}) en los manifiestos YAML de acuerdo al diagrama de arquitectura planteado.

\begin{figure}[h!]
    \centering
    \includegraphics[width=0.85\textwidth]{../diagramas/arquitectura.png}
    \caption{Diagrama de Arquitectura de Microservicios distribuido en el clúster multi-nodo de Kubernetes.}
    \label{fig:arquitectura}
\end{figure}

La distribución física real entre los nodos se configuró de la siguiente manera:
\begin{itemize}
    \item \textbf{Nodo 1 (Control Plane / Host A - \texttt{minikube}):} Aloja al \textbf{API Gateway}, el \textbf{Servicio de Inventario}, el \textbf{Servicio de Pagos} y una de las réplicas del \textbf{Servicio de Reservas}.
    \item \textbf{Nodo 2 (Worker / Host B - \texttt{minikube-m02}):} Aloja a la \textbf{Base de Datos PostgreSQL}, el \textbf{Servicio de Notificaciones} y la segunda réplica del \textbf{Servicio de Reservas}.
\end{itemize}

Esta distribución garantiza la alta disponibilidad del Servicio de Reservas. Si uno de los nodos del clúster falla catastróficamente, el otro nodo cuenta con una réplica activa del servicio capaz de responder al tráfico.

\section{Parte III: Implementación y Pruebas de Resiliencia}

\subsection{Fallo 1: El Inventario Fantasma (Reintentos con Backoff)}
\subsubsection{Justificación del Patrón:}
Se implementó el patrón de \textbf{Reintentos automáticos (Retries)}. Frente a una caída transitoria de un microservicio (causada por un despliegue de Kubernetes o un fallo temporal de red), el sistema no debe devolver un error inmediato al cliente. El backend de reservas realiza hasta 3 reintentos consecutivos espaciados por 1 segundo, permitiendo que Kubernetes auto-recupere el pod de inventario en background y la transacción del cliente finalice con éxito de forma transparente.

\subsubsection{Script de Inyección de Caos (PowerShell):}
El script de caos localiza el pod de inventario y lo destruye de manera forzada sin tiempo de gracia:
\begin{lstlisting}[language=bash, caption=caos/inject\_crash\_inventario.ps1]
# Obtener el nombre del pod de inventario y borrarlo de forma abrupta
$podName = (kubectl get pods -l app=inventario -o jsonpath='{.items[0].metadata.name}' 2>$null)
kubectl delete pod $podName --force --grace-period=0
\end{lstlisting}

\subsection{Fallo 2: La Pasarela Lenta (Circuit Breaker y Timeout)}
\subsubsection{Justificación del Patrón:}
Ante servicios externos lentos (como la pasarela de pagos con latencia de 20s), se aplicó un \textbf{Timeout estricto de 3.0 segundos} combinado con un \textbf{Circuit Breaker en memoria}. Esto evita que el pool de conexiones del Servicio de Reservas se llene de peticiones bloqueadas (previniendo la degradación en cascada). Si se detectan 3 fallos seguidos por timeout, el circuito se abre (\texttt{OPEN}) por 30 segundos, devolviendo un error controlado inmediato al cliente sin intentar llamar a pagos.

\subsubsection{Script de Inyección de Caos (PowerShell):}
El script inyecta la latencia simulada de 20s directamente en el pod de pagos ejecutando un POST interno:
\begin{lstlisting}[language=bash, caption=caos/inject\_latencia\_pagos.ps1]
$podName = (kubectl get pods -l app=pagos -o jsonpath='{.items[0].metadata.name}')
$pyCmd = "import urllib.request; req = urllib.request.Request('http://localhost:8000/config', data=b'{\`"latencia\`": true}', headers={'Content-Type': 'application/json'}); print(urllib.request.urlopen(req).read().decode())"
kubectl exec $podName -- python -c $pyCmd
\end{lstlisting}

\section{Parte V: Análisis Teórico y Diseño de Fallos Complementarios}

\subsection{Fallo 4: Base de Datos Intermitente (Conectividad)}

\subsubsection{Explicación Teórica y Fundamentos Distribuidos}
En un entorno distribuido real, las intermitencias de red son fallos comunes que pueden ser transitorios (ocasionados por pérdida de paquetes, microcortes en el enrutamiento o saturación de sockets de red en Kubernetes) o permanentes. Cuando un microservicio como Reservas intenta escribir en su base de datos durante una de estas fluctuaciones, la falta de una capa tolerante a fallos provoca excepciones inmediatas que anulan las transacciones.

De acuerdo a la teoría de sistemas distribuidos, clasificar los fallos entre \textit{transitorios} (que se resuelven solos al cabo de milisegundos) y \textit{permanentes} es crucial para diseñar la resiliencia. Tratar un error de conectividad transitorio de la base de datos como permanente y cancelar la reserva del cliente degrada innecesariamente la disponibilidad del sistema.

\subsubsection{Relación con la Literatura Académica (MLR)}
Según la investigación académica MLR, el manejo de fallos transitorios debe gobernarse por mecanismos que impidan sobrecargar los servicios internos al recuperarse. Vogels \cite{vogels} introduce la consistencia eventual y explica que los fallos temporales en la persistencia deben gestionarse de forma asíncrona o mediante retornos reintentables no agresivos. 

Si múltiples clientes reintentan la conexión a la base de datos de manera inmediata tras un corte, se produce el fenómeno conocido como \textit{Thundering Herd} (o alud de peticiones), saturando el pool de conexiones disponible.

\subsubsection{Solución Propuesta a Nivel de Producción}
Para mitigar las intermitencias en la base de datos en un entorno productivo de alta escala, se propone un diseño basado en cuatro pilares:
\begin{enumerate}
    \item \textbf{Reintentos con Backoff Exponencial y Jitter:} Los clientes de conexión de base de datos implementan reintentos espaciados exponencialmente ($Delay = Base \times 2^{intento}$), introduciendo una variable aleatoria de ruido o \textit{Jitter}. Esto dispersa las solicitudes de reconexión de manera uniforme a lo largo del tiempo, protegiendo a la base de datos de picos de carga durante su recuperación.
    \item \textbf{PgBouncer (Connection Pooler):} El microservicio de Reservas no abre sockets de forma directa a PostgreSQL. Se interpone PgBouncer en el clúster para reutilizar eficientemente un conjunto limitado de conexiones TCP abiertas a nivel de servidor, mitigando el agotamiento de recursos del sistema operativo.
    \item \textbf{Transactional Outbox Pattern:} En lugar de escribir simultáneamente a la base de datos de reservas y llamar a notificaciones, la transacción de reservas escribe tanto el registro de la compra como un evento en una tabla local de la misma base de datos de forma atómica. Un lector de eventos independiente (*Outbox Reader*) consume esta tabla de manera asíncrona y empuja los mensajes hacia RabbitMQ/Kafka, logrando consistencia eventual y aislando las caídas de conectividad de servicios externos.
    \item \textbf{Llaves de Idempotencia (Idempotency Keys):} Para que los reintentos automáticos enviados por el API Gateway debido a desconexiones transitorias de red no creen reservas duplicadas en base de datos, el cliente debe proveer una llave única (UUID) en las cabeceras HTTP. Reservas valida la existencia de esa llave en un almacén rápido (como Redis con TTL) antes de persistir, asegurando la garantía \textit{Exactly-Once}.
\end{enumerate}

\subsubsection{Pseudocódigo de Conexión Tolerante con Jitter}
\begin{lstlisting}[language=Python, caption=Implementación de reintentos de conexión a PostgreSQL con Backoff y Jitter]
import time
import random

def conectar_con_backoff_y_jitter(max_intentos=5, base_delay=1.0, max_delay=30.0):
    intento = 0
    while intento < max_intentos:
        try:
            # Intentar conexion real a PostgreSQL
            db = conectar_postgres()
            print("Conexion establecida con la base de datos.")
            return db
        except Exception as e:
            intento += 1
            if intento == max_intentos:
                raise Exception("Base de datos indisponible permanentemente.") from e
            
            # Calcular delay exponencial
            delay = min(max_delay, base_delay * (2 ** intento))
            # Introducir Jitter (ruido aleatorio para evitar colisiones)
            sleep_time = random.uniform(0.5 * delay, 1.5 * delay)
            
            print(f"Error de conectividad transitorio ({e}). Reintento {intento}/{max_intentos} en {sleep_time:.2f}s...")
            time.sleep(sleep_time)
\end{lstlisting}

\subsection{Fallo 6: Condición de Carrera (Consistencia)}

\subsubsection{Explicación Teórica y Fundamentos Distribuidos}
La venta concurrente de asientos limitados introduce un reto clásico de consistencia de datos en sistemas distribuidos. Si dos usuarios leen de forma concurrente el estado de disponibilidad del último asiento disponible (\texttt{asiento\_10} en estado disponible = \texttt{True}) y ambos envían la transacción de compra de forma simultánea, se genera una anomalía de lectura sucia o \textit{Lost Update} (actualización perdida), resultando en una inconsistencia catastrófica: la venta duplicada del mismo boleto físico.

Relacionado directamente con el \textbf{Teorema CAP} (Brewer \cite{cap}), este fallo demuestra el trade-off entre la Consistencia Fuerte y la Disponibilidad. Para asegurar que la base de datos de inventario permanezca consistente (C), el sistema debe sacrificar latencia bloqueando las peticiones concurrentes para procesarlas de forma estrictamente secuencial, o bien abortar transacciones concurrentes afectando temporalmente la disponibilidad percibida (A).

\subsubsection{Relación con la Literatura Académica (MLR)}
Kleppmann \cite{kleppmann} detalla en su libro sobre diseño de datos que para prevenir condiciones de carrera concurrentes a nivel de aplicación (donde no basta con usar variables sincronizadas en memoria porque existen múltiples pods balanceando carga), se debe delegar la exclusión mutua al motor de base de datos relacional mediante adecuados niveles de aislamiento transaccional SQL (ANSI/ISO). El aislamiento \textit{Read Committed} no es suficiente porque solo previene lecturas sucias, mas no evita anomalías de actualización perdida en operaciones de lectura-modificación-escritura. Se requiere el nivel de aislamiento \textit{Serializable} o la adquisición explícita de bloqueos sobre los registros de datos.

\subsubsection{Solución Propuesta a Nivel de Producción}
Para garantizar consistencia y prevenir de forma absoluta la doble reserva en producción, se proponen tres alternativas de control de concurrencia:
\begin{enumerate}
    \item \textbf{Bloqueo Pesimista en SQL (Pessimistic Locking):} Utilizar la sintaxis \texttt{SELECT ... FOR UPDATE} al consultar la disponibilidad del asiento en base de datos. Esta instrucción bloquea físicamente la fila del asiento correspondiente en PostgreSQL en un ámbito exclusivo transaccional. Cualquier transacción concurrente que intente leer el estado de esa misma fila se detendrá en espera bloqueada hasta que la primera transacción ejecute un \texttt{COMMIT} o \texttt{ROLLBACK}.
    \item \textbf{Bloqueo Optimista (Optimistic Locking):} Adecuado cuando la contención sobre los asientos es baja. Se agrega una columna \texttt{version} de tipo entero en la tabla de asientos. Al actualizar, la base de datos verifica que la versión no haya cambiado:
    \texttt{UPDATE asientos SET disponible = FALSE, version = version + 1 WHERE id = :id AND version = :version\_leida}.
    Si el número de filas afectadas es cero, significa que otra petición concurrente modificó el asiento primero, haciendo que el backend aborte la transacción local e informe al cliente para que elija otro asiento.
    \item \textbf{Reservas Temporales Distribuidas con Holds en Redis (TTL):} Antes de persistir en PostgreSQL, se adquiere un bloqueo distribuido y rápido sobre la llave del asiento en un almacén en caché en memoria como Redis. La operación se ejecuta de forma atómica (\texttt{SET asiento:id:lock usuario\_id NX EX 600}), bloqueando el asiento por 10 minutos para dar tiempo a que el usuario complete el pago. Una vez pagado, se consolida la transacción en la base de datos relacional y se libera la llave en Redis.
\end{enumerate}

\subsubsection{Pseudocódigo de Reserva Atómica con Bloqueo Pesimista}
\begin{lstlisting}[language=Python, caption=Implementación de reserva consistente utilizando Bloqueo Pesimista SQL]
def procesar_compra_asiento_consistente(db_connection, asiento_id, usuario_id):
    try:
        # Iniciar transaccion explicita
        cursor = db_connection.cursor()
        
        # 1. Adquirir bloqueo exclusivo sobre la fila del asiento (FOR UPDATE)
        # Esto bloquea transacciones concurrentes sobre este ID de asiento especifico
        cursor.execute(
            "SELECT disponible FROM asientos WHERE id = %s FOR UPDATE", 
            (asiento_id,)
        )
        row = cursor.fetchone()
        
        if not row:
            db_connection.rollback()
            return {"status": "error", "message": "El asiento solicitado no existe."}
            
        disponible = row[0]
        
        if disponible:
            # 2. Si esta disponible, cambiar el estado a ocupado de forma inmediata
            cursor.execute(
                "UPDATE asientos SET disponible = FALSE WHERE id = %s", 
                (asiento_id,)
            )
            # 3. Crear el registro en la tabla de reservas
            cursor.execute(
                "INSERT INTO reservas (id, asiento_id, usuario_id, estado) VALUES (%s, %s, %s, 'CONFIRMADA')",
                (generar_uuid(), asiento_id, usuario_id)
            )
            # Confirmar transaccion y liberar el bloqueo exclusivo
            db_connection.commit()
            return {"status": "success", "message": "Reserva exitosa."}
        else:
            # Si ya no esta disponible, liberar el bloqueo con rollback
            db_connection.rollback()
            return {"status": "conflict", "message": "El asiento ya fue vendido a otro usuario."}
            
    except Exception as e:
        db_connection.rollback()
        return {"status": "error", "message": f"Error interno: {e}"}
\end{lstlisting}

\section{Conclusiones}
La realización práctica y análisis de resiliencia en este entorno distribuido multinodo permite consolidar las siguientes conclusiones técnicas:
\begin{enumerate}
    \item \textbf{El fallo es la regla, no la excepción:} En arquitecturas distribuidas de microservicios, las caídas transitorias y la degradación de componentes de terceros son constantes. Asumir una red y un hardware 100\% confiables es un antipatrón. Las aplicaciones deben estar diseñadas para tolerar fallos (\textit{Design for Failure}).
    \item \textbf{Los patrones de resiliencia evitan caídas en cascada:} La integración de Timeouts y el patrón Circuit Breaker impide que un fallo de latencia lenta en un stub no crítico (como pagos o notificaciones) sature las conexiones locales de Reservas, previniendo el colapso de todo el ecosistema.
    \item \textbf{La consistencia distribuida requiere coordinación explícita:} Tal como se analizó en la condición de carrera, para resolver el problema de la doble reserva en alta concurrencia detrás de múltiples réplicas físicas, no es suficiente el uso de memoria local; se debe delegar la exclusión mutua a mecanismos robustos como el bloqueo exclusivo de base de datos relacional o cachés distribuidas atómicas como Redis.
\end{enumerate}
